# 技術設計書

## 1. 技術スタック候補

### 選択肢の比較

| 技術 | Android/iOS | 学習コスト | アニメーション追加 | 開発速度 |
|------|-------------|-----------|------------------|---------|
| **Flutter** | 両対応 | 中 | 可能（Lottie等） | 速い |
| **Unity** | 両対応 | 高 | 容易 | 普通 |
| **React Native** | 両対応 | 低（Web経験者） | やや難 | 速い |
| **Kotlin Multiplatform** | 両対応 | 高 | 可能 | 普通 |

### 決定: Flutter

**理由**:
- UI構築が宣言的で効率的
- このゲームはUIがメイン（3Dグラフィック不要）
- Hot Reloadで開発効率が高い
- 広告SDK（AdMob）、課金（in_app_purchase）のライブラリが充実
- 後からアニメーション追加も可能（Lottie、Rive、CustomPainter + Animation等）

**アニメーションについて**:
- Phase 1ではアニメーションなし（テキスト/スコアボード表示のみ）
- 野球のダイジェストアニメは打席結果のパターンが無限にあり、動的生成が必要
- Flutterでも動的アニメーション生成は可能（CustomPainter + AnimationController）だが、工数が大きいため優先度は低い
- 将来やる場合はFlutterのまま対応可能

## 2. アーキテクチャ

### 2.1 レイヤー構成

```
┌─────────────────────────────────────────┐
│             UI Layer (Flutter)          │
│         画面表示、ユーザー入力           │
├─────────────────────────────────────────┤
│           State Management              │
│      状態管理（Riverpod / Bloc）         │
├─────────────────────────────────────────┤
│            Domain Layer                 │
│    試合ロジック、選手能力計算            │
├─────────────────────────────────────────┤
│            Data Layer                   │
│      ローカルDB、設定、広告SDK           │
└─────────────────────────────────────────┘
```

### 2.2 UI分離の設計方針

後からアニメーションを追加可能にするため、ロジックとUIを明確に分離：

```
Domain Layer（試合ロジック）
    │
    ▼
試合結果データ（イベントログ）
    │
    ├──▶ テキスト表示UI（Phase 1）
    │
    └──▶ アニメーションUI（将来）
```

**イベントログの例**:
```dart
// 1打席の結果を表すデータ
class AtBatResult {
  final Player batter;
  final Player pitcher;
  final int inning;
  final bool isTop;
  final ResultType result; // SINGLE, DOUBLE, STRIKEOUT, HOMERUN, etc.
  final List<BaseRunner> runnersAfter;
}
```

## 3. データ設計

### 3.1 主要エンティティ

```
Team（チーム）
├── id
├── name
├── players: List<Player>
└── settings（スタメン、打順等）

Player（選手）
├── id
├── name
├── position
├── age
├── growthType
├── stats: PlayerStats
└── career（通算成績）

Game（試合）
├── id
├── date
├── homeTeam
├── awayTeam
├── score
├── innings: List<Inning>
└── playerGameStats（試合個人成績）

Season（シーズン）
├── id
├── year
├── games: List<Game>
├── standings（順位表）
└── phase: REGULAR / OFFSEASON

BatterSeasonStats（野手シーズン成績）
├── playerId
├── seasonId
├── games（試合数）
├── atBats（打数）
├── hits（安打）
├── doubles（二塁打）
├── triples（三塁打）
├── homeRuns（本塁打）
├── rbi（打点）
├── runs（得点）
├── stolenBases（盗塁）
├── walks（四球）
├── strikeouts（三振）
└── 算出項目: 打率, 出塁率, 長打率, OPS

PitcherSeasonStats（投手シーズン成績）
├── playerId
├── seasonId
├── games（登板数）
├── starts（先発）
├── wins（勝利）
├── losses（敗北）
├── saves（セーブ）
├── holds（ホールド）
├── inningsPitched（投球回）
├── hitsAllowed（被安打）
├── homeRunsAllowed（被本塁打）
├── strikeouts（奪三振）
├── walks（与四球）
├── earnedRuns（自責点）
└── 算出項目: 防御率, WHIP

CareerStats（通算成績）
├── playerId
├── totalSeasons
├── batterStats（通算野手成績）
└── pitcherStats（通算投手成績）

Title（タイトル記録）
├── seasonId
├── titleType（首位打者, 本塁打王等）
```

### 3.2 ローカルDB

| 候補 | 特徴 |
|------|------|
| **Hive** | 軽量、NoSQL、Flutterと相性良い |
| **SQLite (sqflite)** | RDB、複雑なクエリ可能 |
| **Isar** | 高速、型安全、Hiveの後継的存在 |

**推奨**: Isar または Hive

理由: 試合ログなど大量データを扱うが、複雑なJOINは不要

### 3.3 データ保存方針

| 対象 | 保存内容 | 備考 |
|------|---------|------|
| 最新1試合 | 全球データ（1球ごとのログ） | イニング詳細表示に使用 |
| 過去の試合 | スコア、打席結果、個人成績のみ | 詳細は破棄してデータ量削減 |
| シーズン成績 | 集計済みの成績データ | 順位表、タイトル争い用 |

**理由**: 1試合あたり数万球のデータを全試合保持すると膨大になるため、詳細表示が必要な最新試合のみ全データ保持

## 4. 試合シミュレーション

### 4.1 基本フロー

```
1. 試合開始
2. イニングループ（1〜9回、延長あり）
   ├── 表の攻撃
   │   └── 打席ループ（3アウトまで）
   │       └── 投球ループ（打席決着まで）
   │           ├── 投手が1球投げる
   │           ├── 打者の反応（振る/見逃す）
   │           ├── 結果判定
   │           ├── カウント更新
   │           └── 打席決着判定（三振/四球/インプレー）
   │       └── 走者の進塁処理
   └── 裏の攻撃（同様）
3. 試合終了判定
4. 結果保存
```

### 4.2 Phase 1a の基本確率（現実のプロ野球平均）

| 結果 | 確率目安 | 備考 |
|------|---------|------|
| ストライク（見逃し） | 約15% | |
| ストライク（空振り） | 約10% | |
| ボール | 約35% | |
| ファウル | 約20% | |
| インプレー | 約20% | 打球が飛ぶ |

**インプレー時の内訳**（打球が飛んだ場合）:

| 結果 | 確率目安 | 備考 |
|------|---------|------|
| アウト（ゴロ/フライ） | 約70% | |
| 単打 | 約20% | |
| 二塁打 | 約5% | |
| 三塁打 | 約1% | |
| 本塁打 | 約4% | |

※ これで打率.260程度、三振率18%程度、四球率8%程度になるよう調整

### 4.3 Phase 1a の単純化ルール（走塁）

複雑な走塁は後回しにし、最初は以下の単純ルールで実装:

| 打撃結果 | ランナーの進塁 |
|---------|---------------|
| 単打 | 各ランナー1塁進む（3塁ランナーは生還） |
| 二塁打 | 各ランナー2塁進む（2,3塁ランナーは生還） |
| 三塁打 | 全ランナー生還 |
| 本塁打 | 全員生還（打者含む） |
| 四球/死球 | 押し出しのみ（満塁時に1点） |
| 凡打（ゴロ） | ランナー1塁進む、打者アウト（併殺は後で実装） |
| 凡打（フライ） | ランナー動かず、打者アウト（タッチアップは後で実装） |

**後から拡張する要素**:
- 走力による進塁判定
- 守備力によるアウト/ヒット判定
- 併殺（ダブルプレー）
- タッチアップ
- 盗塁
- エラー

### 4.4 1球ごとの結果判定

```
投球
  ↓
打者の反応
  ├── 見逃し
  │   ├── ストライクゾーン → ストライク（S+1）
  │   └── ボールゾーン → ボール（B+1）
  │
  └── スイング
      ├── 空振り → ストライク（S+1）
      ├── ファウル → ストライク（S+1、ただし2S以上は変わらず）
      └── インプレー（打球）
          ├── 打球種類: ゴロ/フライ/ライナー
          ├── 打球方向: 投手〜ライト（9方向）
          └── 結果: アウト/ヒット（単打〜本塁打）/エラー

カウント判定
  ├── 3ストライク → 三振（打席終了）
  ├── 4ボール → 四球（打席終了）
  └── インプレー → 打球処理（打席終了）
```

### 4.2 打席結果の確率計算（案）

```
基本確率 = 打者能力と投手能力の差から算出

例:
- 打者ミート: 7、投手制球: 5 → 打者有利
- 打者長打力: 8、投手球速: 6 → 長打が出やすい

結果テーブル:
| 結果 | 基本確率 | 能力補正 |
|------|---------|---------|
| 三振 | 15% | 投手変化球↑で上昇 |
| 凡打 | 40% | - |
| 単打 | 20% | 打者ミート↑で上昇 |
| 二塁打 | 10% | 打者長打↑で上昇 |
| 三塁打 | 2% | 打者走力↑で上昇 |
| 本塁打 | 5% | 打者長打↑で上昇 |
| 四球 | 8% | 投手制球↓で上昇 |
```

※ 確率は要調整

## 5. 時間管理（放置系の要）

### 5.1 試合実行タイミング

**方式A: サーバーレス（推奨）**
- アプリ起動時に「前回起動からの経過時間」を計算
- 経過した試合数分をまとめてシミュレーション実行
- サーバー不要、コスト0

**方式B: サーバーあり**
- サーバーで毎日決まった時間に試合実行
- プッシュ通知で結果通知
- サーバー費用が発生

**推奨**: 方式A（サーバーレス）
- 個人開発でサーバー運用は負担
- 「放置」の本質は保てる

### 5.2 実装イメージ

```dart
void onAppResume() {
  final lastPlayedDate = storage.getLastPlayedDate();
  final today = DateTime.now();
  final daysPassed = today.difference(lastPlayedDate).inDays;

  for (int i = 0; i < daysPassed; i++) {
    simulateGame(lastPlayedDate.add(Duration(days: i + 1)));
  }

  storage.setLastPlayedDate(today);
}
```

## 6. 広告・課金

### 6.1 広告

| 種類 | SDK | 表示タイミング |
|------|-----|--------------|
| バナー | AdMob | 画面下部常時 |

### 6.2 課金

| 項目 | 選定 |
|------|------|
| 課金基盤 | RevenueCat（自前サーバー不要） |
| Flutterパッケージ | `purchases_flutter`（RevenueCat公式SDK） |
| 課金形態 | 月額サブスクリプション |
| データ保存 | ローカルのみ（引き継ぎなし） |

**RevenueCatを使う理由**:
- サブスク状態管理を代行（有効/期限切れ/キャンセル等）
- レシート検証を代行
- ダッシュボードで課金履歴確認可能（問い合わせ対応）
- 月間収益$2,500以下は無料
- 自前サーバー構築・運用が不要
